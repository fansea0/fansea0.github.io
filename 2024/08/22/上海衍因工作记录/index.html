兑换码设计

数据库设计

  • privliege_code：兑换码基础信息
  • prvilege_code_cdkey：兑换码本身，一个privliege_code将生成多个prvilege_code_cdkey码
  • privliege_code_redeem：保存兑换码使用过的人

根据提供的数据库表结构图（whiteboard_exported_image-1726629586322-8.png），可以分析出这是一个典型的 兑换码（CDKey）管理系统，设计思路围绕 生成、分发、兑换、权限控制 展开。以下是详细分析：

一、核心表结构与功能

1. privilege_code_cdkey（兑换码基础表）

  • 字段：
    • code_id：唯一标识
    • cdkey：兑换码字符串
    • status：状态（如 normal正常、invalid无效、expired过期）
    • batch_no：批次号（用于批量生成和管理）
    • date_start/date_end：有效期控制
  • 作用：
    存储所有生成的兑换码，管理其生命周期和状态。

2. privilege_code（兑换码规则表）

  • 字段：
    • generate_rule：生成规则（如随机字符长度、前缀后缀）
    • redeem_rule：兑换规则（如限用户、限次数）
    • privilege_rule：权益规则（如赠送会员时长、积分）
    • tracking_rule：追踪规则（如绑定用户IP、设备）
    • permitSubscription：是否允许订阅（如自动续费权益）
  • 作用：
    定义兑换码的生成逻辑、兑换条件和关联权益，与 privilege_code_cdkey 通过 code_id 关联。

3. privilege_code_redeem（兑换记录表）

  • 字段：
    • redeem_user：兑换用户
    • date_redeem：兑换时间
  • 作用：
    记录兑换码的兑换行为，用于审计和防重复兑换。

4. 其他辅助字段

  • 批次管理：
    • batchCount：每批次生成数量
    • genSystem：生成系统（如后台、API）
    • userId/username：操作人（用于责任追溯）
  • 时效控制：
    • dateDuration：动态有效期（如生成后30天内有效）

二、设计思路解析

1. 模块化分层设计

  • 生成层：通过 generate_rule 和 batch_no 实现批量生成，支持不同规则（如随机码、固定模式）。
  • 权益层：privilege_rule 解耦兑换码与具体权益，可灵活配置（如积分、会员、折扣券）。
  • 兑换层：redeem_rule 控制兑换条件（如限新用户、限设备），tracking_rule 增强风控。

2. 状态机管理

  • 兑换码状态流转：
  • 通过 status 字段实现状态控制，避免重复兑换或过期滥用。

3. 批量操作与追溯

  • 批次管理（batch_no）：
    • 支持按批次生成、导出、禁用（如某批次泄露时快速封禁）。
  • 操作日志（genSystem/userId）：
    • 记录生成者和时间，满足合规需求。

4. 动态有效期

  • 支持两种有效期模式：
    • 固定时间范围（date_start/date_end）
    • 相对有效期（dateDuration，从生成时间开始计算）

5. 安全与风控

  • 防刷机制：
    • tracking_rule 可绑定用户/IP/设备，防止恶意兑换。
    • privilege_code_redeem 表记录兑换行为，用于限频分析。
  • 灵活性：
    • remark 字段记录备注（如生成原因、特殊限制）。

三、潜在优化点

  1. 兑换码生成性能

    • 若需高频生成，可引入分布式ID生成器（如雪花算法）替代自增 code_id。

  2. 权益动态加载

    • privilege_rule 可设计为JSON字段，存储结构化权益配置（如 {"type":"vip","days":7}）。

  3. 缓存与查询优化

    • 高频兑换场景下，对 status 和 cdkey 字段建立索引，并缓存热数据。

  4. 国际化扩展

    • 添加 region 字段，支持分区域的兑换码规则（如不同国家兑换码前缀不同）。

四、典型业务流程示例

  1. 生成兑换码

    • 后台配置 privilege_code 规则（如生成1000个7天VIP兑换码）。
    • 调用生成接口，写入 privilege_code_cdkey 表，状态为 enabled。

  2. 用户兑换

    • 用户提交 cdkey，系统校验：
      • 状态是否为 normal
      • 是否在有效期内
      • 是否满足 redeem_rule（如用户未兑换过）
    • 校验通过后：
      • 更新状态为 redeemed
      • 记录 privilege_code_redeem
      • 执行 privilege_rule（如发放VIP权益）。

总结

该设计兼顾灵活性与安全性，通过规则表与数据表解耦，支持复杂的兑换码业务场景（如电商优惠、游戏礼包、会员激活）。如需进一步扩展，可增加黑名单机制或与风控系统联动。
